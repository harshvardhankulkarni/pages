Zoho Creator Design Constraints key factors

ConstraintDesign Impact
No true foreign keysAll relationships are Lookup fields referencing another form's Name field. Archive records instead of deleting to avoid broken references.
No transactionsMulti-step Deluge workflows (e.g. deduct stock + create expense) can leave partial data on failure. Add safety checks before every write and log to a Debug Log form.
Deluge runs per-recordAvoid looping large datasets in on form submit. Use Scheduled Workflows for mass operations: budget alerts, auto-PO closure, KPI refresh.
Subform limitationsSubforms work well for <100 items. For Budget Components and PO Line Items, use separate forms + Add-as-Subform views. Better for reporting.
No dynamic SQLAll aggregation is Map + List in Deluge or Zoho Creator Reports. Prefer Reports over Deluge for performance.
Role-based access is per-formUse form filters on report views or dedicated forms per role where data isolation is needed.
Formula fields can't look upComplex validation (budget checks) must use Deluge workflows on form submit, not formula fields.
Lookup fields cache stale dataStock counts referenced via lookup show old values. Use Formula Lookup (recalculated on load) for live values.

Stock by Warehouse — Separate form Item_Warehouse_Stock with Add-as-Subform:

Source of truth for stock. Every transaction updates this table. Never aggregate on demand.
FieldType
ItemLookup → Inventory_Items Lookup

Deluge approach for Current_Stock:

On every Inventory Transaction (Stock In / Out / Adjustment):
  Update Item_Warehouse_Stock.Current_Stock for the specific warehouse:
    Stock In:  Current_Stock += Quantity
    Stock Out: Current_Stock -= Quantity
    Adjustment: Current_Stock = Adjusted_Qty
  Then recompute Inventory_Item.Current_Stock = SUM of all Item_Warehouse_Stock records for that item

This gives both warehouse-level visibility and item-level totals without on-demand aggregation.

Item Attributes (optional — for variants like size/color):

Create a form Item_Attributes with fields: Item (lookup), Attribute Name (e.g., "Size"), Attribute Value (e.g., "XL"). Linked via Add-as-Subform.

Deluge Workflow (on Submit):

1. Read Allocated_Amount from Budget_Component
2. Read Total_Spent from Budget_Component
3. New_Total = Total_Spent + Expense.Amount
4. If New_Total <= Allocated_Amount:
     Set Status = "Approved"
     Update Budget_Component.Total_Spent = New_Total
     Update Budget_Component status based on % consumed
   Else:
     Set Status = "Overrun-Pending Approval"
     Create record in Budget_Approvals
     Send email notification to Project Manager + Finance Manager

Deluge Workflow (on Submit):

1. Locate Item_Warehouse_Stock record for (Item, Warehouse)
2. Calculate new stock:
   - Stock In:            Current_Stock += Quantity
   - Stock Out:           Current_Stock -= Quantity
   - Adjustment:          Current_Stock = Adjusted_Qty
   - Return to Vendor:   Current_Stock -= Quantity
3. Validate Current_Stock >= 0 before update
4. Update Item_Warehouse_Stock.Current_Stock
5. Recalculate Inventory_Item.Current_Stock = SUM of all warehouse stock
6. If Type = "Stock Out" AND Project is set:
     Create Expense record automatically:
       Amount = Quantity * Rate
       Project = selected Project
       Budget_Component = selected (or auto-map via item category)
       Description = "Auto: Stock consumed — [Item Name] for [Project]"
       Status = "Approved"

Important: The Stock Out → Expense flow is the highest-value automation. It closes the loop between inventory and budget tracking.

PO Lifecycle (matches Zoho Inventory's status model):

Draft → Open → Partially Invoiced → Invoiced → Closed | Cancelled
  • Draft: Being created, not yet submitted
  • Open: Approved and sent to vendor
  • Partially Invoiced: Some line items billed (Phase 2 with Zoho Books)
  • Invoiced: Fully billed (Phase 2)
  • Closed: Items received, PO fulfilled
  • Cancelled: Voided before completion

Deluge Workflow:

  • On Status = "Open": trigger email to vendor with PO PDF
  • On Goods Receipt: update Received_Quantity in line items
  • When Status ≠ "Closed" and all line items Received_Quantity >= Quantity: auto-set Status to "Closed"
  • On Status = "Cancelled": validate no GRN linked to this PO

Dashboard KPIs — Deluge Scheduled Workflow to calculate and store in a Summary_Data form:

  • Total Project Budget = Sum of all active Budget_Plans.Total_Budget_Amount
  • Total Spent = Sum of approved Expenses.Amount
  • Budget Utilization % = Total Spent / Total Project Budget * 100
  • Open PO Value = Sum of POs with Status = "Open"
  • Inventory Value = Sum of (Current_Stock × Purchase_Price) across all items
  • Cost Overruns = Count of Budget_Components where Status = "Exceeded"
  • Low Stock Items = Count of Item_Warehouse_Stock where Current_Stock <= Reorder_Level
  • Active POs = Count of POs where Status IN ("Open", "Partially Invoiced")
  • Pending Receipts = Count of POs where Status = "Open" and past Delivery_Date

Scheduled Workflow (runs daily at midnight):

  • Recalculate all KPI values
  • Check for budget alerts (80%, 90%, 100%)
  • Check for low stock items and send alerts
  • Auto-close POs where all items Received_Quantity >= Quantity and older than 30 days

Phase 1 vs Phase 2 Strategy

Phase 1 (now): All in Zoho Creator — one application, 17 modules, no external integrations. Build forms with Zoho Books & Zoho Inventory compatible field names and statuses for seamless Phase 2 migration.
Phase 2 (future): Zoho Inventory API sync, Zoho Projects task-to-budget linking, Zoho Books accounting push (Invoices → Zoho Books Invoices, Delivery_Challans → Zoho Books DC, Expenses → Zoho Books Expenses, Vendors → Zoho Books Contacts), Zoho Analytics advanced dashboards. The Phase 1 data model is already compatible — migration is field mapping, not a redesign.
Important during Phase 1 build: Do not rename or restructure forms/fields just because it's "Phase 1 only" — future API sync depends on field name alignment.

Implementation Risks & Mitigations watch for these

RiskMitigation
Deluge timeout on large datasetsUse Scheduled Workflows for mass updates. Keep On Submit workflows lightweight — one lookup, one update, no loops.
Negative stockValidate Current_Stock >= Quantity before any Stock Out. Reject transaction with user-visible error message.
Concurrent expense submissionsZoho Creator record-level locking helps. Add Deluge retry pattern: re-read Total_Spent after lock, re-check budget.
Partial workflow failureSafety checks before every write. Log failures to a Deluge_Error_Log form (Form Name, Record ID, Error, Timestamp).
Stale lookup values (stock)Use Formula Lookup (recalculated on load) instead of regular Lookup for any field showing live stock counts.
Overrun notification emails to spamConfigure Zoho Creator email settings. Add sender to contacts. Test with each approver's email.
Formula field performanceKeep aggregation in Deluge, not formula fields. Sum across thousands of records in a Deluge Map/List → write result to a field.
Empty states on reportsEvery report should show "No data yet" message. Dashboard KPI cards show 0 / 0% instead of blank.
Deleting referenced recordsUse archiving (Status = "Inactive") instead of deletion for Projects, Vendors, Items, and Warehouses.

Console Build Instructions team workflow

1. Build Sequence

Follow the 5-phase build order below. Each phase depends on the previous. Don't skip ahead — Phase 1C expenses need Phase 1B budget components.

2. Form Pattern

For each module: Form fields → Validation rules → Deluge workflow → Report/Dashboard → Role permissions. Refer to IMPLEMENTATION_PLAN.md for field-level specs.

3. Seed Data

Create "Main Warehouse" record before any inventory transactions. Create default budget component categories (Labor, Materials, Equipment, Travel, Overhead) for demo projects.

4. Stock Out → Expense

This is the highest-value automation (Workflow #8). Implement it carefully — it closes the loop between inventory and budget. Test with both over-budget and within-budget scenarios.

5. Don't Delete

Archive records (Status = "Inactive") instead of deleting Projects, Vendors, Items, and Warehouses. Zoho Creator has no foreign keys — deleting breaks lookups.

6. Report Issues

Log implementation blockers, design questions, and changes in this repo for team discussion. Update the relevant doc after any design decision changes.
