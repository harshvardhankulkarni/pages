Design Constraints key factors

ConstraintDesign Impact
No true foreign keysAll relationships = Lookup fields. Archive, never delete.
No transactionsSafety checks before every write. Log failures to Debug Log form.
Deluge runs per-recordScheduled Workflows for mass ops. Keep On Submit lightweight.
Subform limitsSeparate forms + Add-as-Subform for >100 items.
No dynamic SQLPrefer Reports over Deluge for aggregation.
Stale lookupsFormula Lookup (recalculated on load) for live stock values.

Risks & Mitigations watch for these

RiskMitigation
Deluge timeout on large datasetsScheduled Workflows for mass ops. Keep On Submit lightweight.
Negative stockValidate Current_Stock ≥ Quantity before Stock Out.
Concurrent expense submissionsRecord locking + Deluge retry: re-read Total_Spent after lock.
Partial workflow failureSafety checks before every write. Log to Deluge_Error_Log form.
Stale lookup valuesFormula Lookup for live stock counts instead of regular Lookup.
Deleting referenced recordsArchive (Status=Inactive) instead of delete.
Formula field performanceKeep aggregation in Deluge, not formula fields.

Console Build Tips team workflow

1. Build Sequence

Follow 6-phase order. Each depends on previous. Refer to IMPLEMENTATION_PLAN.md for field specs.

2. Form Pattern

Fields → Validation → Deluge → Report/Dashboard → Role Permissions. One module at a time.

3. Seed Data

Create "Main Warehouse". Default budget categories: Labour, Materials, Equipment, Travel, Overhead.

4. Stock Out → Expense

Highest-value automation (WF #8). Closes inventory → budget loop. Test both within-budget and over-budget.

5. Don't Delete

Archive (Status=Inactive) Projects, Vendors, Items, Warehouses. Deleting breaks lookups.

6. Log Issues

Blockers, design questions, and changes go in this repo for team discussion.

3 Module Mapping lookups & relationships▸

Zoho Creator uses Lookup fields to relate forms. This is the equivalent of foreign keys in SQL. All relationships in this project are implemented via Lookups.

PatternHow to ImplementExample in This Project
1:1 LookupAdd a Lookup field on the dependent form pointing to the parentBudget_Plans.Project → Projects (one active budget per project)
1:N LookupMultiple child records each look up to a single parentExpenses.Project → Projects (many expenses per project)
Add-as-SubformIn form settings, enable a child form as subformVendor_Contacts inside Vendors; PO_Line_Items inside Purchase_Orders
Denormalized LookupAdd a Lookup to a non-direct parent for report groupingBudget_Components.Project → Projects enables P&L reports to group by project without joins
Formula LookupUse Formula Lookup (not regular Lookup) for calculated values from related recordsItem_Warehouse_Stock stock levels are aggregated by formula into Inventory_Items.Current_Stock

Setting up a Lookup field

  1. Drag a Lookup field onto the form canvas
  2. Click Setup Lookup → select the Application (same app) and Form to look up from
  3. Choose the Display Field (the field shown to users, e.g. Vendor_Name)
  4. Choose Selection Type: Dropdown for small lists (statuses), Popup for large lists (1000+ records)
  5. Enable Criteria to filter lookup values dynamically. Example: In Expenses, Budget_Component lookup should be filtered by Project == current.Project so only components for the selected project appear.

4 Validation Rules data quality▸

Validation rules prevent bad data from being saved. In Zoho Creator, you add them in the form editor under Field Settings → Validation (field-level) or via Form Workflows → On Submit with Deluge throw statements.

TypeWhere to ConfigureExamples in This Project

Field PatternField Properties → Validation → regex patternEmail format, phone format. Custom regex: ^[A-Z]{5}[0-9]{4}[A-Z]{1}$ for PAN validation.
Field RangeField Properties → Validation → min/maxPerformance_Rating must be between 1-5. Tax_Rate between 0-100.
Cross-field ValidationForm Workflow → On Submit → Deluge script with throwEnd Date must be ≥ Start Date. Total of Budget Components ≤ Total Budget Amount. Stock Out quantity ≤ Current Stock + Reserved_Qty. PR items total ≤ available budget.
Duplicate PreventionForm Workflow → On Submit → Deluge getRecords + throwOne active Budget Plan per Project. No duplicate SKUs. No duplicate active BOMs per item.

Deluge throw pattern for cross-field validation

/* Example: Validate End Date >= Start Date (copy into On Submit workflow) */
if (input.End_Date != null && input.Start_Date != null)
{
    if (input.End_Date < input.Start_Date)
        throw "End Date must be on or after the Start Date.";
}

Duplicate check pattern

/* Example: Prevent duplicate active budget plans per project */
criteria = "Project == " + input.Project + " && Status == 'Active'";
existing = zoho.creator.getRecords("budget_tracking", "Budget_Plans", criteria, 1, 1);
if (!existing.isNull() && existing.size() > 0)
    throw "An active Budget Plan already exists for this project.";
5 Workflow Automation Deluge + blueprints▸

Zoho Creator has 3 types of workflows. This project uses all three:

TypeTriggerWhen to Use
Form WorkflowOn Load, On Submit (new), On Submit (edit), On Submit (create or edit), On DeleteReal-time validation, field updates, auto-record creation, notifications. Most automations in this project use On Submit.
Schedule WorkflowCron: daily, hourly, weekly at specific timeBatch operations: budget alerts at 8 AM, low stock alerts, auto-close aged POs, daily KPI refresh.
BlueprintUser-driven stage transitionsApproval flows, multi-stage processes. Used for PR 3-stage approval and PO lifecycle.

5.1 Creating a Form Workflow

  1. Go to Workflows → + Create Workflow → Form Workflow
  2. Name it (e.g. WF_Expense_Budget_Check)
  3. Select the target form (e.g. Expenses)
  4. Choose trigger: On Submit (after the user clicks Save)
  5. Select trigger condition: When a record is created or edited (or choose specific)
  6. In the workflow designer, click + Add Action
  7. Available action types:
    • Set Field Value — Update a field on the current or related record
    • Send Email — Email notifications with record data
    • Add Notification — In-app notification to specific users
    • Update Record — Update fields on the current form
    • Create Record — Create a new record in any form
    • Custom Function — Execute Deluge script (most powerful option)
    • Get Records — Fetch records from any form for processing
  8. For complex logic, use Custom Function and paste Deluge from the build guides
  9. Click Save → Activate toggle to ON

5.2 Creating a Scheduled Workflow

  1. Go to Workflows → + Create Workflow → Schedule
  2. Set name: Sched_Daily_KPI
  3. Set schedule: Daily at 00:00 (or any preferred time)
  4. In the designer, add Custom Function actions
  5. Example: Fetch all active projects, calculate budget % consumed, update dashboard fields, send alerts at 80/90/100%
  6. Enable Activate toggle
  7. Note: Scheduled workflows ignore input. objects — use zoho.creator.getRecords() to fetch data

6 Blueprint Assignment stage-based processes▸

Blueprints enforce structured, stage-based workflows. They are ideal for approval processes and lifecycle management where a record must pass through defined stages.

6.1 When to Use Blueprints in This Project

ProcessStagesTransitions
Purchase Requisition (3-stage approval)Draft → Dept Manager → Finance → Procurement → Approved / RejectedSubmit, Approve, Reject, Revise
Purchase Order lifecycleDraft → Open → Partially Invoiced → Billed → Closed (or Cancelled)Send PO, Receive Goods, Mark Invoiced, Close, Cancel
Expense Approval (budget overrun)Submitted → Overrun-Pending → Approved / RejectedSubmit, Approve, Reject

6.2 Creating a Blueprint

  1. Go to Workflows → Blueprints → + Create Blueprint
  2. Select the form (e.g. Purchase_Requisitions)
  3. Name the blueprint (e.g. BP_PR_Approval)
  4. Set execution: Run this blueprint always (or set condition)
  5. The blueprint canvas opens. Here you define stages and transitions.

6.3 Defining Stages and Transitions

  • Stage = a status value in the process. Drag from the left panel or create new. Each stage maps to a value in the form's Status field.
  • Transition = the action that moves a record between stages. Click between two stages to create a transition line.
  • Configure each transition:
    • Transition Owner — Who can perform this transition? Select users, roles, or user fields.
    • Criteria — Conditions that must be true for the transition to be available (e.g. amount > 10000)
    • Before — Actions executed before the transition (validation checks)
    • After — Actions executed after the transition (notifications, Deluge scripts, field updates)
    • Tooltip — Message shown on the transition button
    • Require Confirmation — Show confirmation dialog before executing

6.4 Associating a Blueprint to a Form

  1. In the Blueprint editor, click Associate (or go to form settings → Blueprint tab)
  2. Select the form layout (standard or mobile)
  3. Map existing status values to blueprint stages (done automatically if names match)
  4. Once associated, the blueprint stages and transitions appear as buttons in form/report views
  5. Users click transition buttons to move records forward. Blueprint enforces the order — users cannot skip stages.

6.5 Blueprint vs Form Workflow: Which to Use?

Use Blueprint WhenUse Form Workflow When
Process has defined stages that records must pass throughLogic runs automatically on form events (submit, edit, delete)
Different users handle different stagesNo manual stage transitions needed
You need to enforce process orderValidation, auto-calculations, real-time updates
Audit trail of who approved/rejected at each stageBackground data sync, mass updates

In this project: Blueprints handle PR approval and PO lifecycle. Form workflows handle all other automations (stock updates, budget checks, notifications).

Set these in Form Settings → Record Sharing after creating roles.

10 Best Practices & Troubleshooting avoid pitfalls▸
AreaPracticeWhy
API NamesUse snake_case. Never rename after go-live. Prefix with form initials for uniqueness.Lookups and Deluge scripts reference API names. Renaming breaks everything.
Lookup fieldsUse Formula Lookup (not Regular Lookup) for values that change in the source recordRegular Lookup caches the value at time of selection. Formula Lookup refreshes on every form load — critical for stock counts, budget balances.
CRUD operationsEvery createRecord, updateRecord, deleteRecord counts as an API callAPI call limits apply based on your plan. Batch operations in Scheduled workflows to reduce calls. One getRecords fetch, then loop with updateRecord.
getRecords limitsMax 200 records per call. Paginate with offset parameter for large datasets.zoho.creator.getRecords("app","form","criteria",page,200). Page starts at 1. Loop with page++ until result size < 200.
Subform limitsMax ~100 rows in a single subform. For >100, use a separate form with Lookup instead.Relevant if PO/Invoice line items exceed 100. Our design uses Add-as-Subform for typical cases.
On Submit vs On Post SubmitUse On Submit for validations. Use On Post Submit for record creation that depends on the current record's ID.In On Submit, input.ID is null for new records. On Post Submit has the ID available.
Delete preventionNever delete records that are referenced by Lookups. Use Status=Inactive/Archived instead.Deleting a record that is a Lookup target breaks all references. This is the #1 cause of data loss in Creator apps.
Deluge timeoutCreator scripts timeout after ~5 minutes. Keep On Submit workflows under 30 seconds. Move batch ops to Scheduled workflows.For mass updates, trigger a Scheduled workflow instead of looping inside On Submit.
Error loggingAlways wrap production Deluge in try/catch and log to an error formWithout logging, failures are invisible. See section 5.5 for the logging pattern.
TestingCreate a separate Test app or use a Dev environment before ProductionZoho Creator doesn't have built-in versioning for data. Test workflows with sample records first.
Auto-number seedsAuto-number fields start at 0001. To change the seed, delete the field and re-add it with the new starting number.Can't change auto-number format after records exist. Plan formats carefully before any data entry.
Mobile formsCheck Mobile Layout in form settings after creating the formForms look different on mobile. Reorder fields, adjust layout for small screens. Enable offline data collection if needed.
Workflow execution logUse Workflows → Execution Log to debug failed workflowsShows which workflows ran, what data was processed, and where errors occurred. Essential for troubleshooting.
