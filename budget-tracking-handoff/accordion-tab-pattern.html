Accordion + Tab Bar Pattern

CSS-only accordion (checkbox toggle) with nested radio-button tab bar. No JavaScript.

Fields tab Relationships tab Workflow tab

User Guide Pattern

Budget Plans define how much money is allocated to a project. Budget Components break that into categories (Labour, Materials, Travel, etc.). Expenses check against these budgets automatically.
One active plan per project: The system prevents creating a second active budget plan for the same project. Close or revise the existing plan first.

Budget Plan Fields

Plan Name Auto-generated
Generated from project name + date.
Project Lookup → Projects
Select the project. One active plan per project enforced by Deluge.
Total Budget Amount Currency
Copied from Project.Total_Approved_Budget or entered manually. Ceiling for all components.
Status Dropdown
Draft → Active (triggers sync to Project) → Revised → Closed.

Budget Component Fields (embedded subform)

Component Name Text
e.g. "Construction Material", "Labour", "Travel", "Equipment".
Allocated Amount Currency
Budget for this category. Sum across all components must not exceed plan's total.
Total Spent Deluge Formula
Aggregated from approved Expenses linked to this component. Updated in real-time.
Consumption % Formula
(Total Spent / Allocated) × 100. Drives the alert status.
Status Auto-computed
Within Budget → 80% Alert → 90% Alert → Exceeded. Updated on every expense submission.
Sum constraint: Σ(Budget_Components.Allocated_Amount) ≤ Budget_Plans.Total_Budget_Amount. Validated on submit via Deluge.

What happens automatically when you interact with Budget Plans and Components.

Status lifecycle

Draft → Active → Revised → Closed

Auto-workflows

On Activate Syncs Total_Budget → Project.Total_Approved_Budget. Enforces unique active plan check.
On Submit Validates Σ(Components) ≤ Total_Budget. Rejects with alert if exceeded.
On Expense Submit Checks new Spent_Amount + Amount vs Allocated. Updates status: 80% → yellow, 90% → orange, 100% → red + overrun flow.
Daily Cron Midnight: refreshes budget KPIs on dashboard, sends alert emails for components at 80/90/100%.

Testing Guide Pattern

Prerequisites: Expense EXP-0002 (Overrun-Pending) must exist. PR REQ-0001 must be approved with PO auto-created. Budget Plan BUD-0001 must be Active.

TC-1D-01: Approve Budget Overrun

  1. Open Budget_Approvals form
  2. Find the auto-created record for expense EXP-0002 (Construction Material overrun)
  3. Set Status = Approved, Notes = "Approved for Phase 1 foundation work"
  4. Submit
Expected: EXP-0002 Status → Approved. Budget_Component "Construction Material" Spent = 27000. Approval No = APPR-0001.

TC-1D-02: Purchase Order — Edit and Open

  1. Open the auto-created PO (Draft) from PR approval
  2. Verify PO_Line_Items: Cement Bag Qty 100 @ 8.50, Steel Rods Qty 50 @ 12.00
  3. Set Vendor = TechSupply Co., Payment Terms = Net 30
  4. Change Status to Open, Submit
Expected: PO-0001 created. On Open → vendor email notification fires.

TC-1D-05: Goods Receipt — Full Receipt

  1. Open Goods_Receipts form, PO = PO-0002
  2. In GRN_Line_Items: Steel Rods, Accepted = 195, Rejected = 5, Warehouse = Main
  3. Status = Open, Submit
Expected: GRN-0001. Stock In auto-created: Steel Rods +195 at Main. Current Stock: 950 + 195 = 1145.

TC-1D-06: Transfer Order — Create and Complete

  1. Open Transfer_Orders, From = Main Warehouse, To = Returns Bay
  2. TO_Line_Items: Cement Bag Qty 20 @ 8.50
  3. Status = Completed, Submit
Expected: TO-0001. Paired Stock Out (Main -20) + Stock In (Returns Bay +20). Main Cement: 510 → 490. Returns Bay: 0 → 20.

Failure conditions and boundary cases to verify during Phase 1D testing.

Budget Approvals

  • Approve without Expense: Try to set Status = Approved without selecting an Expense → alert

Purchase Orders

  • Open without Vendor: Set Status = Open before selecting Vendor → alert "Vendor is required"
  • Zero qty line item: Add line item with Quantity = 0 → should be blocked by validation
  • Cancel with GRN linked: Create GRN against PO, then try Cancel → alert "Cannot cancel PO with linked GRN records"

Goods Receipt

  • Accepted + Rejected > PO Qty: Enter Accepted (150) + Rejected (60) > PO Qty (200) → alert
  • Rejected without reason: Enter Rejected Qty > 0, leave Reason blank → alert
  • PO not Open: Try GRN against a Draft or Cancelled PO → block
  • Accepted but no Warehouse: Accepted Qty > 0, Warehouse blank → alert

Transfer Orders

  • Same warehouse: From = To warehouse → alert "From and To warehouse must be different"
  • Insufficient stock: Transfer qty exceeds available at From → alert
  • Negative stock after transfer: System must prevent Current_Stock < 0 at From warehouse

How It Works

Accordion

Uses a hidden <input type="checkbox"> sibling before the header label. The :checked ~ .accordion-body selector toggles max-height. Only one card open at a time is NOT enforced (use radios if needed).

Tab Bar

Uses hidden <input type="radio"> with shared name attribute per card. Labels toggle radios. :checked ~ #contentN sibling selectors show/hide tab panels. Only one tab active at a time. No JS needed.

Key CSS selectors

.accordion-toggle:checked ~ .accordion-body {
  max-height: 2000px;
}
.tab-radio:checked + .tab-label {
  color: var(--accent);
  border-bottom-color: var(--accent);
}
#tab1:checked ~ #content1 { display: block; }

Tab wiring (HTML order)

<input id="tab1" class="tab-radio" name="t">
<label for="tab1">Fields</label>
<input id="tab2" class="tab-radio" name="t">
<label for="tab2">Relationships</label>
<div id="content1">...</div>
<div id="content2">...</div>

⚠ Radio inputs and content divs must be siblings under same parent for ~ selector to work.
