# Plan 02-02: Zoho Creator Data Service with Caching

## Goal
Build a typed data access layer on top of the Zoho OAuth client. This layer provides 5 aggregated data-fetch methods (mapped to LangChain tools in a later plan) with 5-minute in-memory TTL caching.

  2. `TTLCache.set()` stores value with expiry timestamp; `TTLCache.get()` returns `None` if key is expired or missing
  3. `TTLCache.set()` is thread-safe via `threading.Lock`
  4. Unit test verifies: value stored within TTL is returned, value outside TTL returns None, `invalidate()` removes key, `clear()` removes all keys

<action>

     - `class TTLCache`: `__init__(self)` initializes `self._store: dict[str, tuple[float, Any]] = {}` and `self._lock = threading.Lock()`
     - `set(self, key: str, value: Any, ttl_seconds: int = 300)` — acquires lock, stores `(time.time() + ttl_seconds, value)`
     - `get(self, key: str) -> Any | None` — acquires lock, checks expiry, returns value or None
     - `invalidate(self, key: str)` — removes key if present
     - `clear(self)` — empties store
  2. Write `tests/test_cache.py`:

     - `test_clear` — set two values, clear, assert both gone
  3. Run `python -m pytest tests/test_cache.py -v` — all pass

## Threat Model

<threat_model>
  <threat>
    <id>T-04</id>

    <description>5-minute TTL cache returns stale budget/expense data. A user may make a financial decision based on data that is up to 5 minutes old.</description>
    <mitigation>Cache TTL is configurable via `settings.cache_ttl_seconds`. For time-sensitive queries (e.g., immediate post-expense check), a `bypass_cache` parameter can be added to force fresh Zoho API calls. The cache is in-process only — a service restart clears it.</mitigation>
    <severity>MEDIUM</severity>
    <status>ACCEPTED</status>
  </threat>
  <threat>
    <id>T-05</id>
    <category>ASVS L1 — Data Exposure via Caching</category>
    <description>In-memory cache stores project budget and expense data. If the process memory is compromised, financial data is exposed.</description>
    <mitigation>The cache is ephemeral (process lifetime only). Sensitive data is never written to disk. For production, migrate to an encrypted Redis cache.</mitigation>
    <severity>MEDIUM</severity>
    <status>ACCEPTED</status>
  </threat>
  <threat>
    <id>T-06</id>
    <category>ASVS L1 — Error Information Leakage</category>

    <mitigation>Error messages are sanitized to user-safe strings: "Zoho Creator is temporarily unavailable", "Rate limit exceeded, please wait", "Authentication failed". Internal details are logged at DEBUG level only.</mitigation>
    <severity>MEDIUM</severity>
    <status>ACCEPTED</status>
  </threat>
