  1. `services/anomaly_types.py` defines Pydantic models:
     - `AnomalyDetectionInput`: `project_id: str | None = None`, `severity: str | None = None`
     - `AnomalyDetectionResult`: `anomalies: list[AnomalyFlag]`, `scan_timestamp: str`, `projects_scanned: int`
     - `AnomalyFlag`: `project_id: str`, `component: str`, `expected_range: str`, `actual_value: float`, `severity: str` (low/medium/high), `detected_by: str` (z_score/iqr/both), `recommendation: str`
  2. `services/anomaly.py` exports `class AnomalyDetector`:
     - `__init__(self, data_service: ZohoDataService, z_threshold: float = 2.5, iqr_multiplier: float = 1.5, window: int = 30)`
     - `scan(project_id: str | None = None) -> AnomalyDetectionResult` — main entry point; scans one project or all active projects
     - `_check_expenses(self, project_id: str) -> list[AnomalyFlag]` — fetches expense history, runs both methods per component
     - `_moving_zscore(self, values: list[float], new_value: float, window: int = 30) -> tuple[bool, float]` — returns (is_anomaly, z_score)
     - `_moving_iqr(self, values: list[float], new_value: float, window: int = 30, multiplier: float = 1.5) -> tuple[bool, float, float, float]` — returns (is_anomaly, q1, q3, iqr)
  3. `_moving_zscore`:
     - If `len(values) < window // 2`, return `(False, 0.0)` (insufficient data)
     - Compute rolling mean and std of last `window` values
     - `z = (new_value - mean) / std if std > 0 else 0`
     - Anomaly if `abs(z) > z_threshold`
  4. `_moving_iqr`:
     - If `len(values) < window // 2`, return `(False, 0, 0, 0)`
     - Compute Q1, Q3, IQR over last `window` values
     - Lower bound = Q1 - multiplier * IQR, Upper bound = Q3 + multiplier * IQR
     - Anomaly if value < lower or value > upper
  5. `_check_expenses` ensemble logic:
     - Fetch expenses grouped by component for the project
     - For each component, run both methods
     - Flag as anomaly if EITHER method flags it
     - Severity: `low` if one method flags, `medium` if both flag, `high` if both flag AND `abs(z_score) > 3.5`
     - Generate recommendation based on component type and severity
</acceptance_criteria>

    <category>ASVS L1 — False Negative at Cold Start</category>

    <mitigation>The detector explicitly returns `{"is_anomaly": False, "reason": "insufficient_data"}` during cold start. A global baseline from similar projects is applied (feature-gated behind `settings.use_global_baseline`). The response includes `projects_scanned` and the scan timestamp so users know the system is in warmup mode.</mitigation>
    <severity>MEDIUM</severity>
    <status>ACCEPTED</status>
  </threat>
  <threat>
    <id>T-17</id>
    <category>ASVS L1 — False Positive Flood</category>
    <description>If the z-score threshold is too sensitive (e.g., default 2.5 is the right target per research), routine seasonal purchases could trigger false flags, causing alert fatigue and erosion of user trust.</description>
    <mitigation>Default threshold is z-score 2.5 (< 5% FP rate per research). Thresholds are configurable via Settings. The ensemble method (both z-score AND IQR must not both flag) reduces FP rate. The guardrail layer includes anomaly false positive rate monitoring from AI-SPEC Section 6 offline metrics.</mitigation>
    <severity>MEDIUM</severity>
    <status>ACCEPTED</status>
  </threat>
  <threat>
    <id>T-18</id>
    <category>ASVS L1 — Budget Head Miscoding Anomaly</category>
    <description>An expense posted to the wrong WBS element (e.g., foundation costs under finishing) may produce a false anomaly in one component while hiding a genuine overrun in another.</description>
    <mitigation>The anomaly detector flags anomalies per component. Cross-component transfer detection (`_detect_drift`) checks if the rolling mean of one component decreases while another increases — suggesting miscoding. This is documented as a limitation with a recommendation to verify with the Finance Manager.</mitigation>
    <severity>LOW</severity>
    <status>ACCEPTED</status>
  </threat>
