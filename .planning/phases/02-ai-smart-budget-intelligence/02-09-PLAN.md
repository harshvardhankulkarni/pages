# Plan 02-09: Smart Procurement Recommendation Engine

## Goal
Build a procurement recommendation engine that calculates EOQ (Economic Order Quantity) and threshold-based reorder recommendations using inventory status and consumption history from Zoho Creator. Integrate as a new `recommend_reorder` agent tool for the LangChain agent, accessible via natural language queries and the `/procurement/recommend` REST endpoint.

  2. `services/procurement.py` exports `class ProcurementEngine`:
     - `__init__(self, data_service: ZohoDataService, ordering_cost: float = 500.0, holding_rate: float = 0.2)` — ordering_cost in INR per order, holding_rate as fraction (20% default)
     - `recommend_reorder(item_id: str) -> ProcurementRecommendation` — main entry point for a single item
     - `scan_all() -> ProcurementResult` — scans all inventory items, returns recommendations for items needing reorder
     - `_calc_eoq(self, annual_demand: float, unit_cost: float) -> float` — computes EOQ = sqrt(2 * D * S / (unit_cost * holding_rate)); returns 0 if any input <= 0
     - `_calc_reorder_point(self, avg_consumption: float, lead_time_days: int) -> float` — returns avg_consumption * lead_time_days / 30 (consumption during lead time in months)
     - `_determine_priority(self, current_stock: float, reorder_point: float) -> str` — critical if current_stock <= 0, high if current_stock <= reorder_point, medium if current_stock <= reorder_point * 1.5, low otherwise
     - `_estimate_lead_time(self, procurement_history: dict) -> int` — estimates lead time from PO history: avg days between order date and delivery date; returns configured default (30) if no history
  3. `recommend_reorder(item_id)`:
     - Calls `data_service.get_inventory_status(item_id)` to get current_stock, reorder_point, unit, item_name
     - Calls `data_service.get_procurement_history(item_id, months=12)` to get avg_monthly_consumption and past orders
     - If procurement_history has order records with dates, extracts lead time from order-to-delivery gaps; otherwise uses default 30 days
     - Computes EOQ from annual_demand (= avg_consumption * 12) and unit_cost
     - Computes reorder_point from avg_consumption and lead_time
     - Determines priority from current_stock vs reorder_point
     - If priority is "high" or "critical", recommended_qty = max(EOQ, reorder_point - current_stock + avg_consumption * lead_time_days / 30)
     - If priority is "medium" or "low", recommended_qty = 0 (no immediate reorder needed)
     - recommended_date = today + lead_time_days (ISO format)
     - Builds reasoning string explaining the logic: current stock, reorder point, consumption rate, EOQ calculation, lead time
  4. Error handling:
     - If `get_inventory_status` returns success=False, raises ValueError with the error message
     - If `get_procurement_history` returns success=False, uses default consumption of 0 and logs warning
     - If `avg_consumption <= 0`, returns recommendation with `priority="low"`, `recommended_qty=0`, and reasoning explaining no consumption data
     - If `unit_cost` is None or 0, EOQ is set to None and reasoning explains unit cost unavailable

    <mitigation>The engine uses 12 months of procurement history for consumption averaging, smoothing out monthly variance. Lead time estimation uses actual order-to-delivery gaps from PO history.   </threat>

    <id>T-26</id>

    <description>EOQ assumes constant demand and instant replenishment. For construction materials with bulk discounts, seasonal price variation, or minimum order quantities, EOQ may recommend excess stock tying up >₹2L capital.</description>
    <mitigation>The `reasoning` field explains the EOQ calculation inputs (annual_demand, unit_cost, ordering_cost, holding_rate) so the procurement team can adjust. The holding_rate default of 20% is conservative for construction materials. Priority "low" items do not auto-trigger reorder. Minimum order quantity consideration is a documented limitation for future phases.</mitigation>

    <mitigation>Same mitigation as T-04 — cache TTL is configurable. The endpoint supports a `bypass_cache` parameter for time-sensitive checks (planned future enhancement). The recommendation timestamp is included in the response.</mitigation>
