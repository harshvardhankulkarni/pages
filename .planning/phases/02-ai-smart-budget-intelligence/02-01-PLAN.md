<task id="zoho-oauth-client">
<read_first>
  - 02-01-PLAN.md (Task scaffold-project — must have requirements installed and config.py present)
  - .planning/phases/02-ai-smart-budget-intelligence/02-RESEARCH.md (lines 92-108 — OAuth 2.0 Token Lifecycle; lines 110-143 — API Endpoints)
  - .planning/phases/02-ai-smart-budget-intelligence/02-AI-SPEC.md (lines 227-242 — Project Structure)
</read_first>
<acceptance_criteria>
  1. `zoho_client.py` implements `ZohoClient` class with methods: `get_access_token()`, `refresh_access_token()`, `get_records(report_name, params)`, `get_record(report_name, record_id)`
  2. `get_access_token()` loads refresh token from config, posts to `{accounts_domain}/oauth/v2/token` with `grant_type=refresh_token`, and caches the access token in memory
  3. `refresh_access_token()` is called automatically when HTTP 401 is received from any API call
  4. `get_records()` sends `Authorization: Zoho-oauthtoken {access_token}` header to `{api_domain}/creator/v2.1/data/{owner}/{app}/report/{report_name}`
  5. All API calls implement exponential backoff retry (1s, 2s, 4s) on HTTP 429 and on `requests.exceptions.Timeout`
  6. All API calls raise `ZohoAuthError` on 401 after refresh, `ZohoRateLimitError` on 429 after retries exhausted, `ZohoAPIError` on 5xx
  7. `get_records()` handles cursor-based pagination: reads `record_cursor` response header and follows it
  8. Tests in `tests/test_zoho_client.py` mock `requests.Session` and verify: successful token refresh flow, pagination handling (2 pages of 1 record each), 401-triggered token refresh, 429 retry-with-backoff, error class raising
</acceptance_criteria>
<action>
  1. Write `zoho-ai-service/services/zoho_client.py`:
     - `class ZohoAuthError(Exception)` — raised on authentication failure
     - `class ZohoRateLimitError(Exception)` — raised on rate limit exhaustion
     - `class ZohoAPIError(Exception)` — raised on 5xx server errors
     - `class ZohoClient`:
       - `__init__(self, settings: Settings)` — stores settings, initializes `requests.Session`, sets `_access_token = None`, `_token_expiry = 0`, `_owner` and `_app_name` from settings (with placeholders for now)
       - `_ensure_token(self)` → checks `_token_expiry > time.time()`, if not calls `_refresh_token()`
       - `_refresh_token(self)` → POST to `{accounts_domain}/oauth/v2/token` with `refresh_token`, `client_id`, `client_secret`, `grant_type=refresh_token`; parses JSON response, stores `access_token` and computes `_token_expiry = time.time() + expires_in - 60`
       - `get_records(self, report_name: str, params: dict | None = None) -> list[dict]` — calls `_request_with_retry("GET", f"data/{self._owner}/{self._app_name}/report/{report_name}", params=params)`, handles pagination via `record_cursor`, returns concatenated list of records
       - `get_record(self, report_name: str, record_id: str) -> dict` — calls `_request_with_retry("GET", f"data/{self._owner}/{self._app_name}/report/{report_name}/{record_id}")`
       - `_request_with_retry(self, method, path, **kwargs)` — wraps `_request()` with exponential backoff on 429/Timeout (3 retries: 1s, 2s, 4s) and 401 refresh-and-retry (1 attempt)
       - `_request(self, method, path, **kwargs)` — calls `self._ensure_token()`, sets `Authorization` header, sends request, raises typed exceptions on error status codes
  2. Write `zoho-ai-service/tests/test_zoho_client.py`:
     - Use `pytest` and `unittest.mock.patch("requests.Session.request")`
     - Test `test_successful_get_records` — mock returns 200 with `{"data": [{"id": "1"}], "record_cursor": "cursor2"}` then second call returns `{"data": [{"id": "2"}]}`; assert both records returned

## Threat Model

```xml
<threat_model>
  <threat>
    <id>T-01</id>
    <category>ASVS L1 — Credential Storage</category>
    <description>Zoho OAuth client secret and refresh token stored in plaintext .env file on disk. If .env is committed to git or leaked, an attacker gains persistent Zoho API access.</description>
    <mitigation>.env is added to .gitignore in task scaffold-project. The .env.example template omits real values. Tokens live only in process memory after load.</mitigation>
    <severity>HIGH</severity>
    <status>BLOCKED</status>
    <blocking_note>.gitignore must include .env before any real credentials are written. Add .env to .gitignore in scaffold-project task.</blocking_note>
  </threat>
  <threat>
    <id>T-02</id>
    <category>ASVS L1 — Error Handling</category>
    <description>Zoho API error responses (401, 429, 5xx) may leak internal details (stack traces, Zoho internal IPs) in exception messages.</description>
    <mitigation>Custom exception classes (ZohoAuthError, ZohoRateLimitError, ZohoAPIError) return only user-safe messages. Raw response bodies are logged at DEBUG level only, never exposed to callers.</mitigation>
    <severity>MEDIUM</severity>
    <status>ACCEPTED</status>
  </threat>
  <threat>
    <id>T-03</id>
    <category>ASVS L1 — Rate Limiting Abuse</category>
    <description>Without client-side rate limiting, the service could exceed Zoho Creator API rate limits (~30 req/min), causing temporary IP block.</description>
    <mitigation>zoho_client.py enforces exponential backoff on 429 responses. Future enhancement: add a token-bucket rate limiter if sustained throughput exceeds 30 req/min.</mitigation>
    <severity>MEDIUM</severity>
    <status>ACCEPTED</status>
  </threat>
</threat_model>
```
