# Plan 02-07: Weekly Summary Generation & APScheduler Integration

## Goal
Build the weekly budget summary email generator with APScheduler. The scheduler fires every Monday 8:00 AM IST, fetches all active projects, generates a plain-English budget health summary via the LLM agent, and emails it to project managers. Wire into the FastAPI lifespan.

## Threat Model

```xml
<threat_model>
  <threat>
    <id>T-19</id>
    <category>ASVS L1 — Email Credential Exposure</category>
    <description>SMTP credentials (username/password) stored in .env as plaintext. If the .env file is compromised, an attacker can send emails from the organization's email address.</description>
    <mitigation>SMTP credentials are loaded into `Settings` from `.env` (gitignored). The emailer uses these only in process memory. For production, use OAuth 2.0 for SMTP (XOAUTH2) or an API-key-based service (Resend/SendGrid) to avoid storing passwords.</mitigation>
    <severity>HIGH</severity>
    <status>BLOCKED</status>
    <blocking_note>SMTP password must never be hardcoded in source code. It must be loaded from .env only. The .env file must be in .gitignore. The deployer must create .env manually from .env.example.</blocking_note>
  </threat>
  <threat>
    <id>T-20</id>
    <category>ASVS L1 — Email Content Data Leakage</category>
    <description>Weekly summary emails contain project budget figures sent to the project manager's email address. If the email is intercepted or the PM's inbox is compromised, financial data is exposed.</description>
    <mitigation>Emails are sent via STARTTLS (encrypted in transit). The disclaimer footer is appended. No attachments or raw data dumps — only aggregated summaries. For higher sensitivity, encrypt email body or send via a secure portal link in future phases.</mitigation>
    <severity>MEDIUM</severity>
    <status>ACCEPTED</status>
  </threat>
  <threat>
    <id>T-21</id>
    <category>ASVS L1 — Scheduler Thread Safety</category>
    <description>APScheduler runs jobs in background threads. If a job accesses FastAPI app state concurrently without thread safety, it may crash or produce race conditions.</description>
    <mitigation>SchedulerManager receives dependencies at construction time — it does not access FastAPI app state during job execution. The shared `TTLCache` uses `threading.Lock`. ZohoClient's token refresh is thread-safe via its own lock.</mitigation>
    <severity>LOW</severity>
    <status>ACCEPTED</status>
  </threat>
</threat_model>
```
