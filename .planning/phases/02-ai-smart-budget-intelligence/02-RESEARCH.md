## Zoho Creator API Integration

### Authentication: OAuth 2.0 (Authorization Code flow)

Zoho Creator uses OAuth 2.0 exclusively — no API key option exists [VERIFIED: zoho.com/creator/help/api/v2.1/oauth-overview.html].

**Token Lifecycle:**
1. Register app in Zoho API Console (api-console.zoho.com) → get `client_id` + `client_secret` [VERIFIED]
2. Authorization URL: `https://accounts.zoho.in/oauth/v2/auth?scope=ZohoCreator.report.READ,ZohoCreator.meta.form.READ&client_id=...&response_type=code&redirect_uri=...&access_type=offline` — `access_type=offline` is critical for refresh token [VERIFIED: zoho.com/accounts/protocol/oauth.html]
3. Exchange code for tokens: POST `https://accounts.zoho.in/oauth/v2/token` with `grant_type=authorization_code` [VERIFIED]
4. Access token valid for ~1 hour (`expires_in` field). Refresh token is long-lived (does not expire unless revoked) [VERIFIED]
5. Refresh: POST `https://accounts.zoho.in/oauth/v2/token` with `grant_type=refresh_token` and `refresh_token` [VERIFIED]
6. Include in API calls: `Authorization: Zoho-oauthtoken {access_token}` or `Bearer {access_token}` [VERIFIED]

**Required Scopes:**
- `ZohoCreator.report.READ` — Fetch records from reports (budgets, expenses, inventory) [VERIFIED]
- `ZohoCreator.meta.form.READ` — Get form field metadata (not strictly required but useful) [VERIFIED]

**Data Center:** India = `accounts.zoho.in` for OAuth, `www.zohoapis.in` for API calls. The correct `api_domain` is returned in the access token response — use it rather than hardcoding [VERIFIED].

### API Endpoints

**Base URL:** `https://www.zohoapis.in/creator/v2.1/`

| Endpoint | Method | Purpose | Scope Needed |
|----------|--------|---------|-------------|
| `/data/{owner}/{app}/report/{report}` | GET | Fetch records from a report (max 1000 per request) [VERIFIED] | `ZohoCreator.report.READ` |
| `/data/{owner}/{app}/report/{report}/{id}` | GET | Fetch single record by ID [VERIFIED] | `ZohoCreator.report.READ` |
| `/meta/{owner}/{app}/reports` | GET | List all reports/forms in the app [VERIFIED] | `ZohoCreator.meta.application.READ` |
| `/meta/{owner}/{app}/form/{form}` | GET | Get form field definitions [VERIFIED] | `ZohoCreator.meta.form.READ` |

### Pagination

Max 1000 records per response. Use cursor-based pagination via `record_cursor` header. The response header includes `record_cursor` when more records exist. Pass it in the next request's header [VERIFIED].

### Rate Limits

Zoho Creator does not publish explicit per-minute rate limits publicly. General Zoho API patterns: ~30 requests per minute per application (varies by plan — Standard/Professional/Enterprise). Deluge integration tasks count against external calls limit based on pricing plan [VERIFIED: zoho.com/deluge/help/get-records-v2.1.html]. **Mitigation:** Implement client-side rate limiting with exponential backoff on 429 responses. Cache aggressively (5-min TTL).

## LangChain Agent Design

### Tool Pattern

Use `create_tool_calling_agent` (v0.3.x) with `@tool` decorator for simple tools and `StructuredTool.from_function` for tools needing explicit schema control [VERIFIED: langchain.com/docs].

```python
# Pattern confirmed by official docs and multiple sources
from langchain.tools import StructuredTool

def get_project_expenses_impl(project_id: str, start_date: str, end_date: str) -> dict:
    """Calls zoho_client.get_expenses() with caching and error handling."""
    ...

get_project_expenses = StructuredTool.from_function(
    func=get_project_expenses_impl,
    name="get_project_expenses",
    description="Fetch expense records for a project within a date range (ISO format YYYY-MM-DD). Returns aggregated totals and component breakdown.",
    handle_tool_error=True,
)
```

**Key Rules [VERIFIED: AI-SPEC + langchain docs]:**
- All tools must normalize output to `{"success": bool, "data": Any, "error": Optional[str]}` format
- All tools must cache Zoho responses (5-min TTL)
- All tools must aggregate results before returning to LLM — never expose raw line items
- Tool descriptions must be explicit (agents use them for routing decisions)
- Use `handle_tool_error=True` so errors return to LLM as messages rather than crashing

| Tool | Input | Output | Caching | 
|------|-------|--------|---------|
| `get_project_budget` | project_id: str | Budget plan + component breakdown (aggregated) | 5 min |

### Memory

- **Prototype:** `ConversationBufferMemory` with `trim_messages` to prevent unbounded growth [VERIFIED: AI-SPEC]
- **Production path:** Migrate to LangGraph checkpointing with SQLite/Postgres persistence [HIGH]
- Token budget: 2000 for system prompt + examples, 4000 for budget data, rest for conversation

### Error Handling for Zoho API Calls in Tools

Zoho API calls can fail due to network issues, rate limits, token expiry, or invalid data. The tool should:
1. Wrap API calls in try/except
2. On `requests.exceptions.Timeout` → retry once with exponential backoff
3. On HTTP 401 → refresh OAuth token, retry once
4. On HTTP 429 → wait `Retry-After` header seconds, retry once
5. On HTTP 5xx → return error to LLM ("Zoho Creator is temporarily unavailable")
6. Return `{"success": False, "data": None, "error": "Human-readable message"}` on failure

### Context Window Strategy

Never pass raw line items to LLM. Use tiered aggregation [VERIFIED: AI-SPEC]:
1. Top-level: total budget, total spent, utilization %
2. Component-level: top 5 by variance
3. Drill-down: only when user explicitly asks

## Forecasting Approach

### Method Selection

For a prototype without ML infrastructure, use an ensemble of simple methods:

**Primary: Linear Regression (scikit-learn)** [VERIFIED: multiple sources]
- Fit `sklearn.linear_model.LinearRegression` on monthly expense totals
- X = month index (0, 1, 2...), y = monthly spend
- Extrapolate for 30/60/90-day windows
- Returns trend line + forecast with prediction interval

**Secondary: Moving Average (for smoothing)** [VERIFIED]
- 3-month centered moving average to smooth seasonality
- Useful when trend is noisy or has outliers

**Special handling: Milestone-based billing** [MEDIUM]
- Construction projects bill at milestones (30% foundation, 40% structure, 30% finishing)
- Linear regression will be wrong for milestone-billed projects
- Solution: detect milestone billing from budget plan data, then forecast per milestone rather than per month
- If milestone data isn't available, add a flag: "This forecast assumes linear monthly spend"

### Implementation Plan

```python
import numpy as np
from sklearn.linear_model import LinearRegression

def forecast_burn(monthly_expenses: list[float], horizon_days: int = 30) -> dict:
    """
    Forecast budget burn over horizon_days.
    monthly_expenses: list of monthly totals, most recent last.
    Returns dict with forecast, confidence interval, method used.
    """
    months = np.arange(len(monthly_expenses)).reshape(-1, 1)
    model = LinearRegression()
    model.fit(months, monthly_expenses)
    
    future_months = np.arange(len(monthly_expenses), len(monthly_expenses) + horizon_days/30).reshape(-1, 1)
    forecast = model.predict(future_months)
    
    # Simple confidence: ±2 * residual std deviation
    residuals = monthly_expenses - model.predict(months)
    margin = 2 * np.std(residuals)
    
    return {
        "forecast_total": float(forecast.sum()),
        "confidence_low": float(forecast.sum() - margin),
        "confidence_high": float(forecast.sum() + margin),
        "method": "linear_regression",
        "horizon_days": horizon_days,
    }
```

### Limitations

| Limitation | Impact | Mitigation |
|------------|--------|-----------|
| Lookback bias | One-time large purchase skews baseline [VERIFIED: AI-SPEC] | Use median-based robust regression or clip outliers before fitting |
| Seasonality blindness | Quarterly procurement cycles not captured [VERIFIED: AI-SPEC] | Add month-of-year feature if enough data (>12 months) |
| Milestone timing mismatch | Linear forecast wrong for milestone-billed projects [VERIFIED: AI-SPEC] | Detect milestone billing; use milestone-based forecast |
| Material price escalation | Steel/cement price hikes unaccounted [VERIFIED: AI-SPEC] | Documented limitation; add manual price escalation factor |
| Low data (< 3 months) | Forecast unreliable | Return error: "Insufficient data for forecasting" |
| Data staleness | Forecast based on old data [VERIFIED: AI-SPEC] | Cached data TTL; stale data warning in output |
| Retention money confusion | Withheld 10% retention treated as spent [VERIFIED: AI-SPEC] | Explicitly subtract retention from available budget |

## Anomaly Detection Approach

### Method: Ensemble Moving Z-Score + Moving IQR

Two complementary statistical methods, both suitable for budget data without ML infrastructure [VERIFIED: lenatriestounderstand.com, mcpanalytics.ai]:

**Method 1: Moving Z-Score** [VERIFIED]
- Rolling window (default: 30 days) of historical expense values
- Calculate `z = (current_value - rolling_mean) / rolling_std`
- Flag if `|z| > threshold` (default: 2.5 — adjustable)

**Method 2: Moving IQR** [VERIFIED]
- More robust to non-normal distributions (financial data is often skewed)
- Calculate Q1, Q3, IQR over rolling window
- Flag if value < Q1 - 1.5*IQR or value > Q3 + 1.5*IQR

**Ensemble:** Flag anomaly if EITHER method flags it. Log both methods' scores for tuning.

### Threshold Tuning

| Threshold | Sensitivity | False Positive Rate | Use Case |
|-----------|------------|-------------------|----------|
| z-score > 1.5 | High | High (>10%) | Monitoring, catch all |
| z-score > 2.0 | Medium | Medium (5-10%) | Default initial setting |
| z-score > 2.5 | Low | Low (<5%) [VERIFIED: target] | Production target |
| z-score > 3.0 | Strict | Very low | Only critical anomalies |

**Recommendation:** Start with z-score > 2.5 + IQR 1.5x. Adjust based on feedback.

### Implementation Plan

```python
import numpy as np
from collections import deque

class AnomalyDetector:
    def __init__(self, window: int = 30, z_threshold: float = 2.5, iqr_multiplier: float = 1.5):
        self.window = window
        self.z_threshold = z_threshold
        self.iqr_multiplier = iqr_multiplier
        self.history = deque(maxlen=window)
    
    def check(self, value: float) -> dict:
        """Check if value is anomalous against rolling history."""
        if len(self.history) < self.window // 2:
            self.history.append(value)
            return {"is_anomaly": False, "reason": "insufficient_data"}
        
        arr = np.array(self.history)
        mean, std = np.mean(arr), np.std(arr)
        q1, q3 = np.percentile(arr, [25, 75])
        iqr = q3 - q1
        
        z_score = (value - mean) / std if std > 0 else 0
        iqr_lower = q1 - self.iqr_multiplier * iqr
        iqr_upper = q3 + self.iqr_multiplier * iqr
        
        z_anomaly = abs(z_score) > self.z_threshold
        iqr_anomaly = value < iqr_lower or value > iqr_upper
        
        self.history.append(value)
        
        severity = "low"
        if z_anomaly or iqr_anomaly:
            severity = "medium"
        if (z_anomaly and iqr_anomaly) or abs(z_score) > 3.5:
            severity = "high"
        
        return {
            "is_anomaly": z_anomaly or iqr_anomaly,
            "severity": severity,
            "z_score": float(z_score),
            "expected_range": f"₹{mean - 2*std:.0f} - ₹{mean + 2*std:.0f}",
            "actual_value": value,
        }
```

### Limitations

| Limitation | Impact | Mitigation |
|------------|--------|-----------|
| False positives on seasonal patterns | Routine quarterly purchases flagged as anomalies [VERIFIED: AI-SPEC] | Add seasonal adjustment if >12 months data; exclude from window |
| Slow drift (collective anomaly) | Gradual overspend not caught by point methods | Add trend check: flag if rolling mean drifts >10% over 30 days |
| Contractor payment duplication | Same invoice in two modules appears as double spend [VERIFIED: AI-SPEC] | Cross-reference between expense and POs; dedup check |
| Minimum data requirement | Cannot detect without ~15 data points | Wait for data accumulation; initial period = passive monitoring only |
| Budget head miscoding | Expense posted to wrong WBS element [VERIFIED: AI-SPEC] | Anomaly by component; cross-component transfer detection |
| Cold start | No history on first deployment | Use global baseline from similar projects for first 30 days |

## Scheduled Tasks

### Approach: APScheduler inside FastAPI Lifespan

APScheduler is the right choice for this phase [VERIFIED: sentry.io answers, leapcell.io comparison]:
- No external dependencies (no Redis/RabbitMQ)
- Runs in the same Python process
- CronTrigger for weekly schedule
- Clean FastAPI integration via lifespan context manager
- Persist job state via SQLAlchemyJobStore (optional, for crash recovery)

**Why NOT Celery:** Celery requires Redis/RabbitMQ broker, separate worker processes, more deployment complexity. It's overkill for a single weekly email job. Revisit if many scheduled jobs are needed later [VERIFIED: ceylan.co.at].

### Weekly Summary Job

```python
from contextlib import asynccontextmanager
from apscheduler.schedulers.background import BackgroundScheduler
from apscheduler.triggers.cron import CronTrigger

scheduler = BackgroundScheduler()

def generate_weekly_summaries():
    """Fetch all active projects, generate plain-English budget summary, email PMs."""
    projects = zoho_client.list_projects(status="active")
    for project in projects:
        budget_data = zoho_client.get_budget_breakdown(project["id"])
        summary = llm_chain.invoke({"budget_data": budget_data})
        emailer.send_summary(project["manager_email"], summary)

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup: start scheduler
    trigger = CronTrigger(day_of_week="mon", hour=8, minute=0, timezone="Asia/Kolkata")
    scheduler.add_job(generate_weekly_summaries, trigger, id="weekly_summaries")
    scheduler.start()
    yield
    # Shutdown: stop scheduler
    scheduler.shutdown(wait=False)

app = FastAPI(lifespan=lifespan)
```

### FastAPI x LangChain Integration Pattern

```python
from fastapi import FastAPI, Depends
from pydantic import BaseModel
from langfuse.langchain import CallbackHandler
from agent.agent import agent_executor

class ChatRequest(BaseModel):
    query: str
    chat_history: list[dict] = []
    user_id: str | None = None

class ChatResponse(BaseModel):
    response: str
    sources: list[str] = []

@app.post("/chat", response_model=ChatResponse)
async def chat(req: ChatRequest):
    langfuse_handler = CallbackHandler(
        user_id=req.user_id,
        session_id=req.user_id,
        tags=["nlq-query"],
    )
    result = await agent_executor.ainvoke(
        {"input": req.query, "chat_history": req.chat_history},
        config={"callbacks": [langfuse_handler], "run_name": "nlq-budget-query"},
    )
    return ChatResponse(response=result["output"])
```

### Async Pattern

All LangChain calls should use `ainvoke` (async invoke) to avoid blocking the event loop [VERIFIED: AI-SPEC]. Use `asyncio.to_thread` only for synchronous Zoho API calls that don't have async clients. LangChain's `AgentExecutor` handles `async def` tools natively.
