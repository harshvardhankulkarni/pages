# Plan 02-05: Budget Burn Forecasting Engine

## Goal
Build the forecasting module that generates 30/60/90-day budget burn projections per project using linear regression + moving average ensemble. Wire it into the `/forecast` FastAPI endpoint.

   1. `services/forecast_types.py` defines Pydantic models:
      - `ForecastInput`: `project_id: str`, `horizon_days: int = 30`
      - `ForecastResult`: `project_id: str`, `horizon_days: int`, `forecast_total: float`, `confidence_low: float`, `confidence_high: float`, `method: str`, `monthly_breakdown: list[dict]`, `warning: str | None`, `retention_held: float | None = None`, `available_budget: float | None = None`
      - `ForecastWarning`: an Enum or string literals: `"insufficient_data"`, `"lookback_bias_risk"`, `"milestone_billing_detected"`, `"seasonality_limited"`
  2. `services/forecast.py` exports `class ForecastEngine`:
     - `__init__(self, data_service: ZohoDataService)`
     - `forecast(input: ForecastInput) -> ForecastResult` — main entry point
     - `_linear_regression_forecast(monthly_expenses: list[float], horizon_days: int, project_id: str) -> ForecastResult` — implements sklearn linear regression
     - `_moving_average_forecast(monthly_expenses: list[float], horizon_days: int) -> ForecastResult` — implements 3-month centered moving average
     - `_ensemble_forecast(monthly_expenses: list[float], horizon_days: int, project_id: str) -> ForecastResult` — averages linear regression + moving average
  3. `forecast()` calls the ensemble method by default
  4. Linear regression:
     - X = month index (0, 1, 2...), y = monthly expense totals
     - Must handle edge case where `len(monthly_expenses) < 3` → returns warning `"insufficient_data"` and method = `"none"`
     - Prediction interval: `± 2 * residual_std`
  5. Moving average:
     - 3-month centered moving average
     - Extrapolate: forecast each future month as the average of the last 3 known months
   6. Month calculation: `horizon_days / 30` months projected from most recent data point
   7. `_detect_milestone_billing(self, project_id) -> bool` detection:
      - Scans Budget_Plans records for milestone indicator fields: `Milestone_Name`, `Milestone_Amount`, `Milestone_Due_Date`, or `Payment_Schedule`
      - Returns `True` if any milestone field contains non-null/non-empty values
      - `Payment_Schedule` field is expected as a JSON array of milestone objects: `[{"name": "Foundation", "pct": 30, "due_date": "2026-03-15"}, {"name": "Structure", "pct": 40, "due_date": "2026-06-01"}]`
      - Returns `False` if no milestone-related fields exist in the Budget_Plans schema or data for this project
      - Adds `MILESTONE_BILLING_DETECTED` to `ForecastResult.warning` when milestone billing is detected
   8. Forecast engine handles retention money (10% withheld per AI-SPEC Section 1b failure mode #10):
      - Calls `data_service.get_project_budget(project_id)` to retrieve retention percentage/amount from budget plan
      - If `Retention_Pct` or `Retention_Amount` field is present, computes `retention_held = total_budget * retention_pct / 100` or uses the explicit amount
      - Sets `ForecastResult.retention_held` and `ForecastResult.available_budget = total_budget - retention_held`
      - Adds retention note to `ForecastResult.warning` when `retention_held > 0`

## Threat Model

```xml
<threat_model>
  <threat>
    <id>T-13</id>
    <category>ASVS L1 — Invalid Horizon Input</category>
    <description>An attacker sends `horizon_days=99999` or negative values, causing excessive computation or nonsensical forecasts.</description>
    <mitigation>`horizon_days` validated server-side: only `{30, 60, 90}` accepted. Invalid values return HTTP 400 before any computation starts.</mitigation>
    <severity>LOW</severity>
    <status>ACCEPTED</status>
  </threat>
  <threat>
    <id>T-14</id>
    <category>ASVS L1 — Misleading Forecast Without Warning</category>
    <description>Forecast engine makes a projection with < 3 data points or on milestone-billed projects without warning the user. The user may act on unreliable projections.</description>
    <mitigation>`ForecastResult.warning` field is populated with one of `ForecastWarning` enum values when data quality is insufficient. The API response includes this warning. The agent system prompt instructs the LLM to always mention warnings and confidence intervals.</mitigation>
    <severity>HIGH</severity>
    <status>ACCEPTED</status>
  </threat>
  <threat>
    <id>T-15</id>
    <category>ASVS L1 — Model Skew via Stale Data</category>
    <description>Forecast uses cached expense data that is 5+ minutes old, producing a projection that doesn't reflect the latest spending.</description>
    <mitigation>The `/forecast` endpoint calls `data_service.get_project_expenses` which has a 5-minute cache TTL. The forecast result includes a timestamp. For production, add a `bypass_cache` parameter. The warning field can include `"data_age_seconds"` in future iterations.</mitigation>
    <severity>MEDIUM</severity>
    <status>ACCEPTED</status>
  </threat>
</threat_model>
```
