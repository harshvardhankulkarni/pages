Organization Cycle · End-to-End Flow

Forms & Module Structure

End-to-end Organization Cycle: SALES (Client PO ref) → PRODUCTION → MR → STORE → COSTING → Stock Check → PURCHASE (if needed) → Material Handover → SERVICE (Area→Work→Invoice) → Production GRN → FINANCE → LOGISTICS (DC/Invoice).

3-Layer Hierarchy: SO sells Systems → composed of FGs (System Composition) → each FG formulated from RMs (BOM). MR works at RM level only.

2 · Sales Dual Mode · Project only on Supply+Apply

2A Sales/Work Order Master (SO) Conditional Subforms · System (Supply+Apply) / FG (Supply Only)

Customer details + a line-item subform that changes based on the Sales Type controlling field:
Supply+Apply → sells a SYSTEM (System-code lines) and auto-creates a Project (Stream B root).
Supply Only → sells FG directly (FG-code lines), NO Project created — direct stock sale (Stream A-adjacent).

🎛 Controlling-Field Logic: Sales Type (header dropdown) drives a show/hide rule on the two subforms. Zoho Creator implementation = two subforms on one form, toggled by Sales Type value. Only Supply+Apply fires the SO → Project automation.
⚙ Zoho Creator Implementation Recipe (controlling field = Sales Type):
  1. Form layout: One SO form. Place Subform A (System Lines) and Subform B (FG Lines) as two separate subform fields.
  2. Show/hide rule: Field rule on Sales Type → value = Supply+Apply shows Subform A & hides Subform B; value = Supply Only shows Subform B & hides Subform A (use on user input / on load).
  3. Validation gating: The visible subform must have ≥1 row. Use a Deluge on validate check that targets only the active subform; skip the hidden one so it never blocks submit.
  4. Downstream automation: On SO accept/submit, branch in Deluge — Sales Type == "Supply+Apply" → create Project (Stream B root) + copy Subform A lines into Project Systems subform; == "Supply Only" → NO Project, route to FG dispatch / customer invoice only.
  5. Project-ID guard: MR, MIS, Production, FGHM, Service, Finance, Logistics are reachable only from a Supply+Apply SO. Supply Only FG sales skip them entirely (no MR allocation, no 80% alert).

5 · MR → MIS → Costing Stream B — Project ID

Like Zoho Inventory: warehouse-based tracking with stock in/out/transfer. MR consumes from Store inventory → MIS (stock deduction) → FGHM → Costing valuation. Tagged to Project via Project ID (Stream B).

📌 Per-Project Material Allocation Model (NEW): The Project is the anchor for consumption tracking. Each MR against a Project carries a Material Allocation subform that records, per RM, the Assigned Qty (allocation baseline), its Allocation Ratio (share of total project RM allocation), and an 80% Threshold Alert Flag. Consumption (FG production + Site Manager entries) is measured against this assigned quantity — not global stock. When any allocated RM reaches 80% of its Assigned Qty, the system raises a pop-up + dashboard banner + email to the Project Manager. See Section 5A Material Allocation and the 80% Alert rule.

Backend rule (consumption resolution): Every consumption of this RM against the Project — whether from FG production (BMR / RM Consumption) or a Site Manager consumption entry — increments Consumed Qty on the matching MR Allocation line (matched by Project ID + Item Code). The system never deducts from a generic pool for project-tagged material. See Consumption Flow (flow.html).

80% Alert: When Consumption % crosses 80% for any line with the Alert Flag ON, the system fires a pop-up + dashboard banner + email to the Project Manager. See 80% Consumption Alert.

⚠ 80% Consumption Alert — Per Allocated Material (KEY REQUIREMENT)

Trigger: For each RM line in the MR Material Allocation subform, when Consumed Qty ÷ Assigned Qty × 100 ≥ 80% AND the 80% Threshold Alert Flag is ON, raise a warning.

Example: Material X assigned 10 units → alert fires when X has been consumed 8 units (80%). Material Y (20) → at 16 units. Material Z (30) → at 24 units.

Notifications fired: (1) In-app pop-up to the user saving the consumption entry; (2) Dashboard banner on the Project dashboard (and Store/Service dashboards); (3) Email to the Project Manager (and Project Coordinator). Banner escalates to "Allocation Exhausted" at 100%.

Automation Rules

  • Manual initiation (always): No auto-creation triggers. Finance must manually click "New Credit Note" and fill the form.
  • On PO lookup: Auto-fetch Supplier details, Project ID. Line items are NOT auto-filled from PO — Finance manually selects items to credit.
  • On QC lookup (Scenario 1): Auto-fetch rejected items list with quantities into credit line items as draft. Finance reviews, edits if needed, and confirms.
  • On GRN lookup (Scenario 2): Auto-fetch short quantity (PO Qty − GRN Received Qty) into credit line items as draft. Finance reviews and confirms.
  • Status: Draft → Approved: Requires Account & Finance senior approval. Records timestamp and approver.
  • Status: Approved → Credited: Marks the credit as settled against supplier. Updates Supplier Outstanding summary.
  • On Credit: Notification to Purchase dept and Supplier manager about the credit adjustment.
  • PO Link: Credit Note number and total credited amount recorded on the linked PO for audit trail.
  • Cancellation: If cancelled, no financial impact. Cancelled notes are retained for audit.
