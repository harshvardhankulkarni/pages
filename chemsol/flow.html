Organization Cycle · Interactive Demo

Process of Productivity

Two-Stream Architecture: Stream A (Stock Procurement): PURCHASE → STOCK (no Project ID). Stream B (Project-Centric): MASTER DATA → SO → PROJECT → MR (Costing Gate) → PRODUCTION → INVENTORY (Project ID flows downstream).

3-Layer Hierarchy: SO sells Systems (EP01) → composed of FGs (System Composition) → each FG formulated from RMs (BOM). MR works at RM level only.

Stream A PR → PO → GRN → QC → STOCK ⇔ Stream B MASTER DATA → SO → PROJECT → MR (COSTING GATE) → PRODUCTION → INVENTORY

⚙ How FG is Made from RM

Example: To make 100 Kg of FG-EP-001 (Epoxy Base Coat): Consume 85 Kg Epoxy Resin + 12 Kg Hardener + 3 Kg Pigment via BMR → yields 100 Kg FG. The BOM defines these ratios. The MR allocates them per project. MIS issues them from store. BMR records the actual consumption. FGHM hands over the finished FG.

🔍 FG Creation Traceability

Each FG unit recorded in FGHM can be traced back to its BMR batch (via BMR No.), which consumed specific RM batches (RM Batch No. in BMR line items). This provides full batch-level traceability from RM procurement → production → FG handover.

Data Flow Key

🔗 Project ID flows through ALL Stream B forms 🔗 Reference No PR→PO→GRN→QC 🔗 AutoFetch Item details, UOM, rates
Pro tip: Every Stream B transactional form carries the Project ID — that's how the system stays linked. You never enter data in isolation.
Project as Consumption Anchor: MR carries a Material Allocation subform (per-RM Assigned Qty + Allocation Ratio + 80% Alert Flag). All consumption resolves to the project-assigned RM, not a generic pool, and raises an 80% alert as each material nears its allocation.
3-Layer Product Hierarchy: System (sold on SO — EP01) → FG (produced in production) → RM (raw materials via BOM — MR works at this level). MR requisitions RM only. BMR consumes RM to produce FG.
