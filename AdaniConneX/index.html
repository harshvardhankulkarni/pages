Overview

AdaniConneX is implementing ZohoCRM Enterprise to automate Data Center Opportunity sales workflows across lead management, sales cycle, contracting, and capacity management. This plan covers the full lifecycle from opportunity creation through SOF Contracted (100%), including cross-team approvals, pricing validation, document management, dependent system integrations, and contract renewals.

Taxation (Client Script)

IGST vs CGST+SGST determined by comparing Account Billing State vs Data Center Site State. A Client Script fires on onChange of either field to set tax type:

  • Intra-state (same state) = CGST+SGST
  • Inter-state (different states) = IGST

Taxation applied at line level in both Deals and Quotes modules.

India-First Compliance

7A. EdgeOS Portal Integration

Direction: ZohoCRM → EdgeOS (push every 15 minutes)

Trigger: Stage change to SOF Contracted + scheduler every 15 min for batch sync / retry.

Data to Sync
Field Source in Zoho
Opportunity Reference IDOpportunity_Reference_ID
Customer NameAccount_Name.Account_Name
Site IDSite_ID
Contracted Power (KW)SUM(Colocation_Pricing.Power_Density_KW)
Contract Status"Active" / "Expired" / "Cancelled"
Contract Expiry DateContract_Expiry_Date
Deluge Pseudocode (Custom Function: sync_sof_to_edgeos)
// Step 1: Query all SOF Contracted opportunities
criteria = "ACX_Sales_Stage == 'SOF Contracted'"
oppRecords = zoho.crm.getRecords("Data_Center_Opportunity", criteria);

// Step 2: Iterate and build payload
for each opp in oppRecords:
    // Join Colocation Pricing subform
    pricingLines = opp.get("Colocation_Pricing");
    totalPowerKW = 0;
    for each line in pricingLines:
        totalPowerKW += line.get("Power_Density_KW");
    end

    // Build JSON payload
    payload = Map();
    payload.put("opportunityRefId", opp.get("Opportunity_Reference_ID"));
    payload.put("customerName", opp.get("Account_Name").get("Account_Name"));
    payload.put("siteId", opp.get("Site_ID"));
    payload.put("contractedPowerKW", totalPowerKW);
    payload.put("contractStatus", "Active");
    payload.put("contractExpiryDate",
        opp.get("Contract_Expiry_Date").toString("yyyy-MM-dd"));

    // POST to EdgeOS API
    headerParams = Map();
    headerParams.put("Authorization", "Bearer " + edgeosApiKey);
    headerParams.put("Content-Type", "application/json");

    response = invokeUrl(
        url: "https://edgeos.adaniconnex.com/api/v1/sync-opportunity",
        type: POST,
        headers: headerParams,
        body: payload.toString()
    );

    // Update sync status
    updateRecord = Map();
    if response.getResponseCode() == 200:
        updateRecord.put("EdgeOS_Sync_Status", "Synced");
    else:
        updateRecord.put("EdgeOS_Sync_Status", "Sync Failed");
    end
    zoho.crm.updateRecord("Data_Center_Opportunity",
        opp.get("id"), updateRecord);
end
What EdgeOS Must Provide
  • REST API endpoint (e.g., POST /api/v1/sync-opportunity)
  • Authentication mechanism (API key in header, or OAuth2 client credentials)
  • Network accessibility: EdgeOS endpoint reachable from Zoho servers
  • Upsert logic: create customer+site+power limit if not exists, update if exists
  • Expected payload schema, response codes, error messages documentation
Architecture Options
Option Pros Cons
A. Custom Function → Webhook → EdgeOS APIReal-time push, no middlewareEdgeOS needs public API endpoint
B. Scheduler + Deluge → JSON export → FTP/SFTPNo EdgeOS API needed, simpler15-min delay, file management overhead
C. Zoho → Middleware (Zapier/Workato) → EdgeOSNo coding, visual connectorMonthly cost, external dependency

Recommended: Option A if EdgeOS has a REST API. Option B if EdgeOS does not.

Sync Status Tracking

Field on Opportunity: EdgeOS_Sync_Status — Synced (HTTP 200), Sync Failed (retry on next scheduler run), Pending (awaiting first sync).

Option A — File-based (Recommended Initially)
ZohoCRM Custom Function "generateSAPExport":
1. Trigger: Workflow rule "On Deployed Date update" calls function
2. Query: Get opportunity + all pricing subforms
   - Colocation Pricing (space + power lines)
   - Additional DC Products & Services (ancillary items)
3. Map to SAP structure:
   - Customer master from Account (billing address, GST, PAN)
   - Material codes from product-to-SAP mapping table
   - Quantities from pricing subforms
   - Prices from MRC/OTC fields
4. Generate CSV with fixed-width columns per SAP spec:
   - HEADER: Customer, Material, Qty, UoM, Price, Currency, Tax...
   - DETAIL: one row per product line
   - TRAILER: total amounts, record count
5. Write file to SFTP server that SAP polls
6. Update status: "Billing Exported"

SAP Import Job (by SAP team):
1. Polls SFTP folder periodically
2. Validates file (checksum, record count)
3. Creates billing documents via BAPI
4. Writes ACK file back: opportunityRefId, status, SAP doc number

Zoho Polling (scheduler every 30 min):
1. Checks SFTP for ACK files
2. On Confirmed → "Billing Confirmed" + SAP_Document_Number
3. On Failed → "Billing Failed" + error reason
Option B — Direct API (Future State)

Custom Function calls SAP RFC/REST API (BAPI_BILLINGDOCUMENT_CREATEMULTIPLE or OData service) synchronously. Immediate confirmation. Requires SAP connector, network connectivity, and API user with billing authorization.

What SAP Must Provide
  • BAPI/RFC or REST API for billing document creation
  • SFTP drop folder details (host, port, credentials, path) — for Option A
  • Web service endpoint details — for Option B
  • Product-to-material-code mapping (AdaniConneX product name → SAP material number)
  • SAP team POC for import job development

Status tracking: Field on Opportunity: SAP_Billing_Status — Billing Pending → Billing Exported → Billing Confirmed / Billing Failed. Plus SAP_Document_Number field for the billing document reference.

Open: EdgeOS API availability and auth mechanism. SAP import file format (CSV/IDoc/API). Existing middleware availability. EdgeOS sync scope — only SOF Contracted or also SOF MSA with future-dated start?

Daily Workflow Comparison

Before (Current State)

  1. Sales creates opportunity
  2. Manually types the stage from memory
  3. Uploads documents when remembered (no enforcement)
  4. Emails Pre-Sales / Commercials asking for approval
  5. Tracks contract dates & renewals in personal Excel
  6. No one knows if pricing is complete
  7. No automatic handoff to Service Delivery
  8. EdgeOS & SAP teams wait for manual email

After (Target State)

  1. Sales creates opportunity with all required fields
  2. Enters pricing in structured subforms
  3. Uploads PO/SOF, clicks "Move to SOF MSA"
  4. Blueprint validates → Pre-Sales auto-notified
  5. Pre-Sales approves → Commercials auto-notified
  6. Commercials approves → stage moves automatically
  7. Notification fires to Service Delivery + Sales Heads
  8. EdgeOS synced, SAP export triggered automatically

Deluge Script Constraints

  • Execution: max 5 minutes per Custom Function run
  • API calls: limited per-day per-org (check ZohoCRM Enterprise limits)
  • Response size: max 1MB for API responses
  • No complex recursion, no external JS/library imports

Blueprint Limitations

  • Forward transitions only — backward needs separate rejection flow
  • Blueprint cannot invoke Custom Functions directly; use Workflow Rules as bridge

Integration Reliability

  • EdgeOS: 3 retry attempts before Sync Failed; alert IT on failure
  • SAP SFTP: connection health monitoring; dead-letter alerts on failed exports
  • Both integrations need status tracking fields on Opportunity record

Data Migration Risks

  • Existing free-text pricing data cannot auto-map to structured subforms — manual cleanup
  • Existing Stage transitions may not match new Blueprint rules — bulk update needed
  • Existing Documents may not be linked to correct Opportunities — relinking effort

User Adoption Risks

  • Sales team may resist mandatory pricing subforms (currently free-text)
  • Approval workflow adds latency if approvers are slow to respond
  • Mitigate new mandatory fields with phased rollout + training

Security

  • EdgeOS API keys: store in Zoho Connections Manager, never hardcode in Deluge
  • SAP SFTP credentials: store in secure parameters
  • Role hierarchy must be fully implemented before go-live
